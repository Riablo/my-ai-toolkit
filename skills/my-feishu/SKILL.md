# My Feishu

使用 `lark-cli im +messages-send` 发送飞书群消息。这个 skill 只做一件事：把用户要发的内容发送到白名单里的群聊，并在失败时给出最小但有效的排障路径。

## CLI 定位

- 优先使用 PATH 中的 `lark-cli`
- 先运行 `lark-cli im +messages-send --help`，确认命令可用
- 若命令不存在或帮助都无法执行，停止发送并明确告诉用户先安装或修复 `lark-cli`

## 允许发送的群

只允许向下表里的群发送消息：

| 名字或别名 | chat_id |
| --- | --- |
| 测试群 | `oc_9fcef6c3a03f113b03bd0db4dc3c1c67` |
| 工具群、前端群 | `oc_20c59f6f6d49c6fc2400fe0f81ff5e01` |
| Chat2Vision群、C2V群 | `oc_3a798bc91b0381ed94e520905bd798a8` |
| 720产品线群、上线群 | `oc_17f8165f3112c2cf82c829552b5752bb` |
| AI群 | `oc_c26554daaa8e7ca5ce09931943e9f16a` |

匹配规则：

1. 若用户给的是 `chat_id`，必须与表中某一项 `chat_id` 完全一致；否则取消发送。
2. 若用户给的是群名，只在“名字或别名”列中做保守匹配。优先精确匹配，其次再做明显的一对一模糊匹配。
3. 若匹配结果为 0 个或多于 1 个，必须取消发送，不要猜测，不要尝试搜索新群。
4. 不允许发送到白名单外的任何群，即使 `lark-cli` 本身可以发送也不行。

## 发送前检查

每次真实发送前都检查下面几项，不能静默跳过：

1. `lark-cli im +messages-send --help` 能正常执行。
2. 目标群已经通过上面的白名单校验。
3. 消息内容非空。
4. 这次任务是“发群消息”，不是私聊；不要使用 `--user-id`。

如果用户只说“发到飞书”但没给目标群或内容，先补齐最少必要信息后再执行。

## 默认发送方式

- 默认使用纯文本：`--text`
- 如果只是简短的一句话、一个短提醒或简单通知，优先使用 `--text`
- 如果消息内容较多，或者本身带有分段、列表、标题等结构，优先使用 `--markdown`，让展示更清晰
- 可以适度加入少量 emoji 提升可读性，但不要堆砌，不要影响正文理解
- 若文本包含多行、缩进或代码块，优先使用 shell 的 `$'...'` 形式保留换行
- 这个 skill 当前只封装群文本消息；不要扩展成图片、文件、卡片、私聊或群管理

常用命令：

```bash
lark-cli im +messages-send --chat-id <chat_id> --text "你好，这是一条测试消息"
```

```bash
lark-cli im +messages-send --chat-id <chat_id> --text $'第一行\n第二行'
```

```bash
lark-cli im +messages-send --chat-id <chat_id> --markdown $'## 通知\n\n- 已完成\n- 请查收'
```

## 推荐流程

1. 识别用户给的是群名、别名还是 `chat_id`。
2. 用白名单解析出唯一 `chat_id`；若解析失败，停止。
3. 确认消息内容。
4. 根据消息复杂度选择发送方式：短句或短提醒用 `--text`；较长内容或结构化内容用 `--markdown`；如有需要，可加少量合适的 emoji。
5. 执行 `lark-cli im +messages-send --chat-id <chat_id> --text ...` 或 `--markdown ...`。
6. 成功后，向用户转述发送到哪个群、发送了什么类型的内容。
7. 失败后，按“失败排障”处理，不要盲目重试。

## 失败排障

常见排障顺序：

1. 先看是不是白名单问题：
   - `chat_id` 不在表里，直接判定为无效目标，不发送。
   - 群名匹配不到或不唯一，直接停止，让用户明确目标群。
2. 再看 CLI 是否可用：
   - 运行 `lark-cli im +messages-send --help`
   - 如果连 help 都失败，提示用户先修复本机 `lark-cli`
3. 再看配置是否已初始化：
   - 运行 `lark-cli config show`
   - 若明显未初始化，引导用户运行 `lark-cli config init --new` 或按其现有配置方式修复
4. 再看是否是 user 身份授权问题：
   - `auth login` 永远是给 user 身份登录，不是给 bot 用的
   - 如需 user 身份授权，先发起：

```bash
lark-cli auth login --domain im --no-wait
```

   - 然后用返回的 `device_code` 继续轮询完成授权：

```bash
lark-cli auth login --device-code <device_code>
```

5. 再看是否是 bot 身份或 scope 问题：
   - 如果报的是权限不足、机器人不可见、机器人不在群里、scope 不足，不要让用户执行 `auth login` 解决 bot 问题
   - 明确告诉用户去飞书开放平台检查该应用的 IM 发送权限、可见范围和 bot 是否已加入对应群

## 输出转述

- 成功时：先用自然语言告诉用户消息已发到哪个白名单群，再补充使用的是哪个 `chat_id`
- 失败时：先明确“未发送成功”，再说明是白名单拦截、CLI 故障、未登录，还是权限问题
- 除非用户明确要原始输出，不要整段倾倒 CLI JSON

## 边界

- 不要搜索或发现新群
- 不要修改白名单
- 不要发送私聊
- 不要管理群成员、历史消息、文件、卡片或其他 IM 能力
- 当用户需求超出“向白名单群发消息”时，明确说明这个 skill 不处理
