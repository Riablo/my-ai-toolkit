# Reflection Checklist

创建、更新或 review skill 时，按这个顺序检查。

## 先看是否真的需要这个 skill

- 没有这个 skill，模型会稳定做错吗
- 这是不是一句全局规则就能解决的问题
- 这部分知识是不是变化太快，不适合固化进 skill

## 再看 description

- 是否主要在写“何时触发”
- 是否覆盖了真实用户会说的话
- 是否过长
- 是否把执行细节、默认数量、保存路径之类的信息塞进 description
- 是否会和相邻 skill 抢路由

## 再看正文

- 有没有把 help 或 README 复述进去
- 有没有写模型本来就知道的常识
- 主文件里真正高价值的信息是否足够密集
- 有没有把“默认动作”和“不要自动做什么”讲清楚

## 再看 gotcha

- 最容易误触发或做错的地方有没有写
- 失败时最该补的是不是 gotcha，而不是更长流程
- 有没有写出危险操作、互斥参数、写本机状态、敏感信息处理

## 再看结构

- 主文件是否过胖
- 条件性重内容是否应该拆去 `references/`
- 重复执行且确定性强的逻辑是否应该拆去 `scripts/`

## 再看维护性

- 这次改动是否影响 README
- 这次改动是否需要补路由样例或 eval 线索
- 是不是在把 skill 写成一次性文档，而不是可迭代的上下文包
