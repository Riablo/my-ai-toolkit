# Routing And Gotchas

这份 reference 专门用来检查两个最容易出问题的面：路由触发器，以及 gotcha 密度。

## Description 怎么看

一个好的 `description` 应该尽量满足：

- 优先描述用户会怎么说
- 更像“当用户提到……时使用”
- 简短
- 不总结 workflow
- 不塞默认参数、输出目录、并行策略、具体文件名

## Description 的常见坏信号

- 一上来就在解释 skill 的能力清单
- 包含太多“默认会做什么”
- 把 references、脚本名、保存路径都写进去
- 语义太宽，像“处理所有和 X 有关的任务”
- 完全没有相邻边界

## Gotcha 应该长什么样

高价值 gotcha 通常是：

- 模型容易误判的边界
- 用户没明确授权时不能自动做的事
- 参数互斥、危险默认值、隐式副作用
- 输出最容易被误读的地方
- 真实失败案例中反复出现的问题

## Gotcha 的坏信号

- 只写成功路径
- 失败时只会补更多命令示例
- 把 gotcha 写成抽象口号，没有落到具体动作
- 没区分“常见失败”和“只是罕见理论情况”

## 这类问题怎么修

如果路由太宽：

- 收紧触发词
- 加相邻边界
- 删除 workflow 描述

如果正文太胖：

- 删掉常识和手册内容
- 把重内容拆去 `references/`

如果 gotcha 太少：

- 从最近一次失败回放里抽一条负例
- 优先补“不要自动做什么”和“哪里最容易误判”
