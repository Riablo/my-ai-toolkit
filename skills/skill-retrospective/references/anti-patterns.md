# Anti-Patterns

这个文件记录创建和维护 skill 时最容易重复出现的坏味道，优先覆盖这个仓库已经踩过的坑。

## 1. 把 description 写成功能介绍

坏例子：

- “使用某某 CLI 查询天气、检查配置并返回 JSON”
- “这是一个用来帮助 AI 管理 Obsidian 笔记的 skill”

问题：

- 它在解释 skill 做什么，而不是何时触发
- 路由词太少，模型更难在真实用户表达里命中
- 还会把索引预算浪费在执行细节上

更好的方向：

- 写用户会怎么说
- 写相邻需求的边界
- 尽量短，不总结工作流

## 2. 把 SKILL.md 写成 README

坏例子：

- 整段复制 `-h`
- 把所有子命令示例都列出来
- 写一堆模型本来就知道的 bash / git / CLI 常识

问题：

- token 成本高
- 真正有价值的 gotcha 被埋掉
- 一旦 CLI 变更，skill 比 help 更容易过时

## 3. 只写主流程，不写容易做错的地方

坏例子：

- 只说“运行这个命令”
- 不说什么情况下不要自动执行
- 不说哪些参数互斥、哪些操作会写本机状态、哪些输出容易误读

问题：

- 模型最需要的是失败边界，而不是顺风局说明书

## 4. 所有内容都塞进主文件

坏例子：

- 主文件同时放路由、教程、枚举表、模板、长清单、维护手册

问题：

- 主文件一旦触发，整段都要进上下文
- 条件性内容无法渐进加载

更好的方向：

- 主文件只保留入口规则、核心流程、关键 gotcha
- 重内容拆到 `references/`

## 5. 没有相邻边界意识

坏例子：

- 新 skill 用特别宽的描述，把“创建 skill”“审查 skill”“修改 README”“整理文档”全吃进去
- 不区分普通 skill、CLI skill、repo 内 skill、通用 skill

问题：

- 一个新 skill 会把别的 skill 路由打乱
- 表面只改了一个 skill，实际上让整个集合更不稳定

## 6. 用“补更多说明”代替“补 gotcha”

坏例子：

- 每次失败都追加一段更长的流程说明

更好的方向：

- 先记录这次失败属于哪类
- 优先补负例、边界、禁止事项和路由样例

## 7. 改了 description 却没有补样例

问题：

- description 是最敏感的路由面
- 小改词也可能影响别的 skill

最少也应该补：

- 一组应该触发的表达
- 一组不该触发的相邻表达

## 8. 为了“完整”而增加无用文件

坏例子：

- skill 目录里塞 README、CHANGELOG、QUICK_REFERENCE

问题：

- 看起来整齐，其实增加噪音
- 模型并不需要人类项目文档式的包装
