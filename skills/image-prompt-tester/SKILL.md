# Image Prompt Tester

用于测试网上看到的图片生成提示词。核心目标不是复述提示词，而是基于原始提示词生成若干个有代表性的变体，实际生成图片，并把图片和每张图使用的最终提示词一起保存和回传。

## 触发条件

当用户出现以下意图时使用：

- “生图测试”“测试这个提示词”“测试这个 prompt”“试几张图”
- “用东京、北京、首尔测试这个提示词”
- “这个图片提示词效果怎么样，帮我生成几张看看”
- 用户给出一段图片生成提示词或场景描述，并要求实际生成图片

如果用户只是想优化、翻译、解释提示词，但没有要求生成图片，不要自动生成；可以只给文本建议。

## 主代理职责：区分后台会话与子代理

这个工作流可能耗时并产生多张图片。触发后要先判断用户是否期望“当前聊天窗口不被阻塞”。

关键机制：`delegate_task` 会派生子代理并隔离上下文，但它是同步工具调用；父 agent 会等待子代理全部完成后才继续当前回复。因此 `delegate_task` 适合减少主上下文污染、并行子任务，但不等于后台任务。若用户来自 Telegram/Discord 等消息客户端，并明确或隐含希望继续聊天不受阻塞，应优先使用 Hermes 的 `/background` 机制，或提醒用户用 `/background 生图测试：...` 发起任务。`/background` 会启动独立后台会话，完成后再把结果发回原会话。

推荐做法：

1. 主代理解析用户输入，得到：
   - 原始提示词或场景描述
   - 用户指定的测试内容列表（如东京、北京、首尔）
   - 用户指定的生成张数；未指定时默认 3 张
   - 是否需要把结果发回消息客户端
   - 用户是否要求主聊天继续可用/不要阻塞
2. 若缺少“要测试的提示词/描述”这种关键输入，再向用户询问；其它轻微不确定性直接按默认值执行。
3. 如果用户在消息客户端请求非阻塞执行，优先建议或使用 `/background` 重新发起完整任务；不要把 `delegate_task` 描述成能解锁当前会话的后台机制。
4. 如果是在当前 turn 内执行，或用户未要求非阻塞，则可调用 `delegate_task`。需要生成多张图时，优先使用 `delegate_task(tasks=[...])` 的 batch 模式，把每个最终提示词/变体拆成一个子任务并行生成；不要默认把 3 张图交给单个子代理串行生成。子代理至少需要 `image_gen` 工具；为了保存和校验文件，通常同时给 `terminal`、`file` 工具集。
5. 子代理完成后，主代理只汇总输出目录、图片清单、最终提示词和回传状态。

如果当前环境不能使用子代理或后台会话，才由主代理自己执行同样流程。不要只告诉用户“我会生成”，必须实际调用图片生成工具。

## 提示词语言要求

无论用户给出的原始提示词是英文、中文还是中英混合，最终用于 `image_generate` 的每个变体提示词都必须使用简体中文撰写。

- 不要把英文原文直接作为最终生图提示词。
- 如果原文是英文，先理解并改写为中文变体，再生成图片。
- 专有名词、品牌名、模型名、镜头型号等确实不宜翻译的内容可以保留原文；其余画面描述、风格、光线、构图、材质、限制条件都应写成自然的简体中文。
- negative prompt 或“不要出现...”类限制也应尽量转成中文表述。
- `01-*.txt`、`prompts.json` 和 `manifest.md` 中记录的“最终提示词”也要和实际调用图片生成工具的提示词一致，使用简体中文。

### 子代理任务模板

委派时把下面这些要求写进子代理 prompt 中，确保子代理没有上下文也能完成：

```text
你要执行 Image Prompt Tester 工作流。

输入：
- 原始提示词/描述：...
- 用户指定测试内容：...（如无则写“未指定”）
- 生成数量：...（未指定则 3）
- 输出根目录：~/Downloads
- 最终提示词语言：简体中文（无论原始输入是什么语言）
- 当前会话是否需要消息端回传：...

要求：
1. 不要直接使用原始提示词本身生成图片；必须先生成变体。
2. 每个变体的最终生图提示词必须用简体中文写成；即使原始 prompt 是英文，也要先翻译并本地化改写成中文提示词后再调用图片生成工具。
3. 如果用户指定了测试内容，每个内容生成一个变体，并让提示词中的地点、文化、气候、建筑、语言、道具等相关描述一起合理调整。
4. 如果用户未指定测试内容，自己识别原始提示词中适合随机替换的槽位，默认生成 3 个差异明显但保留核心风格的变体。
5. 每个变体调用图片生成工具生成 1 张图；多张图要尽量并行发起，优先用 `delegate_task(tasks=[...])` 一图一子任务，或在支持并行工具调用的环境中并行调用 `image_generate`。
6. 在 ~/Downloads 下创建一个新的测试文件夹，保存：原始输入、每个最终提示词、每张图片、manifest。
7. 校验图片文件存在且非空。
8. 输出每张图的本地路径和最终提示词；如果是 Telegram/Discord 等客户端会话，还要让主回复能带回图片和提示词。
```

## 变体生成规则

### 用户指定测试内容时

用户给出列表时，按列表逐项生成，不再额外随机扩展，除非用户明确要求更多。例如：

- “用东京、北京、首尔测试” -> 生成 3 张
- “用上海和纽约各两张” -> 生成 4 张
- “换成猫、狐狸、机器人测试” -> 生成 3 张

替换时不要机械替换单个词，要做语义联动：

- 地点：同步调整地标、街道、建筑风格、招牌语言、天气、城市氛围、人群穿着等。
- 时代：同步调整服装、交通工具、材料、技术物件、色彩和摄影质感。
- 主体：同步调整动作、体型、材质、环境互动和道具。
- 风格：同步调整媒介、画面纹理、光影、构图和色彩，而不是只追加一个风格词。

### 用户未指定测试内容时

根据原始提示词自动找可替换槽位，默认生成 3 个变体。优先选择能明显影响画面、但不会破坏核心 prompt 的部分：

- 城市/国家/自然场景
- 主体类型、职业、物种、载具或物品
- 时间：季节、时代、昼夜、天气
- 视觉风格：摄影、电影、插画、像素、海报、油画、3D 等
- 色彩/材质/灯光/镜头
- 文化元素、服饰、招牌语言、背景道具

保留这些内容，除非它们与新变体冲突：

- 原始提示词的核心构图和主题
- 质量要求、比例要求、镜头/光线设定
- negative prompt 或“不要出现...”类限制
- 用户明确要求固定的角色、产品、品牌、颜色或风格

### 输入不是完整 prompt 时

如果用户给的是描述而不是完整 prompt，也可以生成候选提示词再生图。做法：

1. 从描述中抽取核心场景、主体、风格和用途。
2. 默认生成 3 个完整提示词变体。
3. 每个变体都要足够可直接用于图片生成，并在 manifest 中标记为“由描述扩展”。

## 图片生成

使用当前可用的图片生成工具（如 `image_generate`）。不要假设具体模型名或模型参数；只传入最终提示词和合适的比例。

比例选择：

- 用户明确要求比例时照做。
- prompt 明确是头像、人物立绘、海报、手机壁纸时优先 portrait。
- prompt 明确是图标、贴纸、logo、封面实验时优先 square。
- prompt 明确是风景、城市、电影剧照、横版构图时优先 landscape。
- 不确定时默认 landscape。

每张图只生成 1 次；生成失败时可轻微调整违反限制或过于模糊的部分并重试 1 次。若仍失败，在 manifest 中记录失败原因，不要伪造图片路径。

## 输出文件规范

每一组测试在 `~/Downloads` 下新建一个文件夹。命名建议：

```text
~/Downloads/image-prompt-test-YYYYMMDD-HHMMSS-<slug>/
```

`<slug>` 从原始提示词的核心主题或用户给出的测试主题中提取，转为短英文/拼音/安全字符；无法提取时用 `prompt`。

文件结构建议：

```text
image-prompt-test-YYYYMMDD-HHMMSS-<slug>/
├── original-prompt.txt
├── manifest.md
├── prompts.json
├── 01-<variant-slug>.txt
├── 01-<variant-slug>.<ext>
├── 02-<variant-slug>.txt
├── 02-<variant-slug>.<ext>
└── 03-<variant-slug>.txt
```

保存规则：

- `original-prompt.txt`：保存用户原始输入，不要改写。
- `01-*.txt`：保存该图片实际使用的最终提示词；最终提示词必须是简体中文。
- 图片文件：若图片工具返回本地路径，复制到测试文件夹；若返回 URL，用 `curl -L` 下载到测试文件夹；若无法下载，至少在 manifest 中记录 URL。
- `prompts.json`：结构化记录原始输入、变体、图片路径/URL、状态和错误原因；最终提示词字段用简体中文。
- `manifest.md`：人类可读索引，包含每张图的编号、变体说明、最终提示词、图片路径/URL。

保存后必须校验本地图片文件存在且大小大于 0；可用 `test -s`、`file`、`python3` 或其它轻量方式检查。

## 消息客户端回传

如果任务来自 Telegram、Discord、Slack 等能展示图片的客户端：

1. 仍然先保存完整结果到 `~/Downloads/...`。
2. 最终回复中逐张包含图片和对应提示词，让客户端能直接看到图片。
3. 如果当前网关支持 Markdown 图片展示，可用 `![01 变体名](本地路径或URL)` 后跟提示词。
4. 如果只能发送文本，则发送输出目录、图片路径/URL 和提示词；不要使用只适用于其它平台的 `MEDIA:` 标签，除非当前运行环境明确要求。

在 CLI 会话中，最终只需给出输出目录和文件清单；不要输出 `MEDIA:` 标签。

## 最终回复格式

完成后简洁汇报：

```text
已完成生图测试。
输出目录：~/Downloads/image-prompt-test-YYYYMMDD-HHMMSS-xxx

1. <变体名>
   图片：<path-or-url>
   提示词：<最终提示词>

2. ...
```

如果某张图失败，明确标注失败，并说明重试过什么或下一步建议。不要把未生成成功的项目说成成功。

## 质量检查清单

结束前确认：

- 没有直接用原始提示词生成图片。
- 用户指定的测试内容都覆盖了；未指定时默认 3 个变体。
- 每张图都有对应的最终提示词文件。
- 每个最终生图提示词都是简体中文；原始英文 prompt 没有被直接拿去生图。
- 结果保存在 `~/Downloads` 的新建文件夹内。
- 本地图片已校验存在且非空，或 manifest 中记录了无法下载的 URL/失败原因。
- 消息客户端请求已尽量回传图片和提示词。
