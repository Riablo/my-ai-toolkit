# yun720

使用已经安装并配置好的 `yun720` CLI 操作 720 云开放平台。这个 skill 只记录工作流和注意事项；命令细节以 CLI 当前帮助为准。

## 核心原则

- 不要凭记忆猜参数。每次执行前先查看 CLI 帮助，至少运行：

```bash
yun720 -h
```

- 遇到子命令、参数、模板、状态值、路径格式不确定时，继续查看对应帮助，例如：

```bash
yun720 config -h
yun720 pano -h
yun720 pano upload -h
yun720 pano status -h
yun720 pano wait -h
yun720 tour -h
yun720 tour create -h
```

- CLI 已安装，配置也已配置好。优先直接使用 PATH 中的 `yun720`。
- 源码仓库不是运行时依赖；不要假设 `/Users/cz/Work/720yun-cli` 会长期存在。
- 真实执行前先确认用户意图、输入图片/目录/已有 panoId/作品名是否足够；缺关键参数时再询问。

## 能力范围

根据 `yun720 -h` 和子命令帮助动态选择命令。当前 CLI 大致覆盖：

- `config`：查看或设置 720 云 API 配置。
- `pano`：上传全景图片素材、查询素材制作状态、等待制作完成。
- `tour`：创建 720 漫游作品；可以使用已有 panoId，也可以从本地图片一体化上传、等待、创建。

不要把这里当作完整命令文档；参数细节以 `yun720 ... -h` 为准。

## 配置与依赖检查

虽然用户已配置好，但执行真实请求前仍要做轻量检查：

1. 运行 `yun720 config show`，确认 CLI 能读到配置；输出里 CLIENT_KEY 通常会脱敏，直接转述即可。
2. 如果配置缺失或 CLI 报错，提示用户或协助运行 `yun720 config set`；不要继续上传或创建作品。
3. 若 CLI 提示缺少系统依赖（如 `curl`、`jq`、`openssl`、图片尺寸工具等），先安装或引导修复，再重试。

## 用户意图到命令的路由

先查看帮助，再按下面思路选择：

- 用户要“查看配置/环境/凭证是否生效”：用 `yun720 config show`。
- 用户已有本地全景图片或目录，只想上传成素材：用 `yun720 pano upload ...`；需要减少交互时结合帮助确认是否可用 `-y/--yes`。
- 用户给出 taskId，想看制作结果：用 `yun720 pano status <taskId>`。
- 用户要等素材制作完成：用 `yun720 pano wait <taskId>`，必要时设置 interval/timeout。
- 用户已有 panoId，想创建作品：用 `yun720 tour create --name ... --panos ...`。
- 用户给出本地图片/目录并想“一步创建作品”：用 `yun720 tour create --name ... --from ...`。
- 用户提到模板、水印、等待时间等选项：先看对应 `-h`，再添加参数。

## 图片输入要求

执行上传或创建前，先确认素材符合 CLI 帮助中的要求。当前常见要求包括：

- 支持 `.jpg/.jpeg`。
- 2:1 等距柱状全景图适合单张上传。
- 六面体素材通常是 6 张 1:1 图片并按 CLI 帮助要求命名。
- 文件大小、比例、命名规则以 `yun720 pano upload -h` 当前输出为准。

如果用户给的是 PNG/WebP 或其它格式，先转换成 JPG/JPEG 再交给 CLI。转换后用 `sips`、`identify` 或其它可用工具检查尺寸比例。

## “生成图片”创建素材或作品

当用户要求“生成图片”来创建 720 云素材或作品，而不是提供现成图片地址/本地路径时：

1. 使用当前图片生成工具（配置为 gpt-image-2 系列模型）先生成图片。
2. 生成时使用 2:1 的 equirectangular 360° panorama 提示词。把 `{填写场景}` 和 `{填写风格}` 替换成用户需求；如果用户没有给风格，选一个朴素、适合全景展示的默认风格。
3. 保存生成结果到本地工作目录，必要时转换为 `.jpg`。
4. 检查图片尺寸是否接近 2:1；若不是，先说明限制并尽量重新生成或后处理到 2:1，再上传。
5. 再根据用户目标运行 `yun720 pano upload` 或 `yun720 tour create --from`。

生成全景图片时使用这个提示词模板：

```text
使用 gpt-image-2 模型生成一张 2:1 比例的 360 度全景图片，适合放入全景播放器中旋转观看。

这必须是 equirectangular 360° panorama，不是普通宽幅风景图。请确保图片左右两侧边缘能够无缝衔接，左边缘和右边缘在地平线、光照、天空、地面、建筑、山脉、水面、纹理和色彩上都要自然连续，避免旋转时出现明显接缝、断裂、错位或重复痕迹。

不要把主要主体放在左右边缘交界处。保持地平线水平且连续。不要出现文字、水印、边框、UI 或播放器界面。

场景内容：
{填写场景}

风格：
{填写风格}

输出一张真正可用于 360 度全景播放器的无缝全景图。
```

注意：图片生成模型不一定完美保证 360° 无缝和 2:1。上传前务必检查尺寸；如果视觉质量或比例明显不符合全景播放器用途，先告知用户并重新生成，不要盲目上传。

## 执行方式

- macOS 自带 `/bin/bash` 是 3.2，若遇到 `${p,,}: bad substitution`，说明 yun720 脚本用了 Bash 4+ 语法。优先用 Homebrew Bash 运行：`PATH=/opt/homebrew/bin:$PATH yun720 ...`。
- 长流程（上传、等待制作、创建作品）可能耗时较久；用足够长的 timeout 或后台进程监控。
- 上传前 CLI 可能要求确认；若任务明确、用户已经要求执行，可结合帮助使用跳过确认的参数。
- 执行命令时保留完整 stdout/stderr，方便提取 taskId、panoId、作品 ID、预览链接和错误原因。
- 不要把 API 密钥、完整配置文件或敏感 token 原样展示给用户。

## 输出转述

执行后先给结论，再列关键结果：

- 上传素材：成功/失败、taskId、panoId、是否已进入制作队列。
- 查询/等待：任务状态、成功或失败原因、下一步建议。
- 创建作品：作品 ID、tid、预览链接。
- 生成图片参与流程：本地图片路径、尺寸、是否已上传/创建。

除非用户明确要求调试，否则不要整段倾倒原始 JSON；提取最有用字段即可。失败时保留 CLI 报错中的关键行，并说明你接下来会如何修复或需要用户补充什么。
